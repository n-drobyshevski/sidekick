# Wiz Sidekick — Google Apps Script rebuild

A full rebuild of the Streamlit Wiz vulnerability dashboard as a Google Apps Script
web app: OS-level CVEs on host workloads, severity breakdowns, and MTTR / SLA
remediation analytics over a persistent scan history — with **Google Sheets** as the
durable ledger and **Drive** for raw scan archives.

The Python `wiz_dashboard/domain/` layer is the behavioral spec: the TypeScript
`src/domain/` modules are 1:1 ports verified against golden fixtures generated by
running the *original* Python code (see `test/export_fixtures.py`,
`test/export_ledger_fixtures.py`). 117 vitest cases replay them, including full
persist → delete-replay → compaction flows against the SQLite implementation.

## Architecture

| Layer | Where | Notes |
| --- | --- | --- |
| Pure domain logic | `src/domain/` | severity, vuln_key identity (sha1), MTTR/SLA metrics (pandas-interpolation quantiles), reconcile, domain rules, trend, compaction, **ledgerCore/maintenance** (in-memory persist/delete/checkpoint machinery) |
| Storage | `src/server/` | `sheetsDb` (header-mapped tabs), `archiveStore` (Drive gzip JSON), `ledgerStore` (commit-record writes + journals), `settingsStore`, `historyStore`, `jobsStore` |
| Wiz client | `src/server/wizClient.ts` | OAuth client-credentials + GraphQL on UrlFetchApp; `wizQuery.ts` is generated verbatim from `os_vulns.py` |
| Scan jobs | `src/server/scanJobs.ts` | resumable page walk (6-min limit) via one-shot trigger continuation |
| Web app | `src/client/` | hash-routed SPA served by `doGet`; Chart.js 4 bundled into `js_app.html` (no CDN); DESIGN.md system in `styles.css` |

### How SQLite transactions were replaced

- **Commit-record ordering**: the `scans` row is appended **last**. A scan whose row
  is absent from the `scans` tab does not exist, regardless of partial writes.
- **Journals**: before every mutation the prior state is gzipped to
  `backups/` in Drive; `recoverIfNeeded()` restores it when a job died mid-write.
- **Single-flight**: every mutation runs inside `LockService` script lock; scan /
  delete / compact jobs are serialized through the `jobs` tab.
- **Stats-identity gate**: compaction recomputes MTTR + trend before/after in memory
  and refuses to apply if anything changed (port of the SQLite rollback gate).

### Deliberate schema deviations (documented, intentional)

- **`observations` live in Drive, not a tab** (`obs/obs-<scan_id>.json.gz`).
  One row per finding per scan would cross the 10M-cell spreadsheet ceiling in weeks
  at 100k findings/day. Only two readers exist (previous-scan severity counts,
  compaction accounting) — each reads one file.
- **Compaction checkpoints live in Drive** (`checkpoints/`), referenced by
  `compactions.checkpoint_ref` — the blob exceeds the 50k-char cell cap.
- **Raw archives are multi-part** (`scans/<scan_id>/page-NNNN.json.gz`), written
  page-by-page during the fetch so a 100k-finding payload never sits in memory.
- **Incremental scans archive the merged slim set** as their replayable payload
  (the delta's full raw pages are kept as `delta-page-*` files). Replay correctness
  is preserved — reconcile reads only slim fields — but the raw JSON export of an
  incremental scan carries slim records, not full nodes.
- **Dry-run mode** uses the bundled sample payloads with a deterministic
  resolve-one-per-scan evolution — a light stand-in for `data/demo.py`.

## Setup (one-time)

1. `npm install && npm run check` (typecheck + tests + build).
2. Create an Apps Script project (`clasp create --type webapp` or use an existing
   script id) and put its id in `.clasp.json`.
3. `npx clasp login`, then `npm run push` (builds and pushes `dist/`).
4. In the GAS editor, run **`setup()`** once. It creates the "Wiz Sidekick Ledger"
   spreadsheet, the `wiz-sidekick` Drive folder skeleton, the daily scan trigger
   (05:00 UTC), and records their ids in Script Properties.
5. Set the remaining Script Properties by hand (never committed anywhere):
   - `WIZ_API_URL` — your tenant GraphQL endpoint, e.g. `https://api.<region>.app.wiz.io/graphql`
   - Authentication — pick **one**:
     - `WIZ_API_TOKEN` — a Wiz service-account bearer token, used directly (simplest;
       matches `js_request_exemple.js`). It **expires and is not auto-refreshed**, so the
       daily scan will 401 once it lapses — good for quick validation, less so for the
       standing schedule.
     - `WIZ_CLIENT_ID` / `WIZ_CLIENT_SECRET` — a Wiz service account; the app performs the
       OAuth client-credentials exchange and auto-refreshes the token. Preferred for the
       daily scan. (`WIZ_AUTH_URL` defaults to `https://auth.app.wiz.io/oauth/token`.)
     - If both are set, `WIZ_API_TOKEN` wins.
   - `WIZ_PROJECT_ID_V2` — the Wiz project id the scan is scoped to
   Without credentials the app runs in dry-run mode over sample data.
6. Deploy: **Deploy → New deployment → Web app** (execute as you, access: domain),
   or `npm run deploy`.

## Development

```
npm run check        # tsc + vitest + esbuild
npm test             # vitest only
npm run push         # check + clasp push
```

- Fixtures: regenerate after touching the Python domain layer —
  `python gas/test/export_fixtures.py && python gas/test/export_ledger_fixtures.py`
  (needs `pip install pandas`).
- `src/server/wizQuery.ts` / `sampleData.ts` are **generated** — edit
  `gas/test/extract_query.py` / `extract_samples.py` instead.
- `dist/entry.js` is hand-written (GAS needs top-level `doGet`/`api_*`/`trigger_*`
  globals) and git-tracked; the rest of `dist/` is build output.

## Manual verification checklist (GAS-only behavior)

Things node tests cannot cover — verify after the first deployment:

- [ ] `setup()` provisions tabs/folders/trigger idempotently (run it twice).
- [ ] OAuth token fetch against the real tenant; a scan lands rows in `scans`,
      `vuln_ledger`, and files under `scans/<id>/` + `obs/`.
- [ ] Trigger continuation: temporarily set `FIRST_STEP_BUDGET_MS` low, confirm a
      multi-hop scan completes and the one-shot trigger cleans itself up.
- [ ] Kill an execution mid-persist (editor Stop) → next write logs the journal
      recovery and the `scans` tab shows no phantom scan.
- [ ] Web app at `/exec`: charts render (Chart.js bundled, no CDN fetch), Blob CSV download works in the IFRAME
      sandbox, the drill-down sheet traps focus and closes on ESC, hash deep links
      (`#/overview?sev=CRITICAL&q=log4j`) restore filters.
- [ ] Quick refresh on a tenant that rejects `updatedAt` filters surfaces the
      "run a full scan" message (WizDeltaFilterError path).
- [ ] Delete of a sealed scan refuses; delete of an unsealed scan rebuilds and the
      MTTR page numbers match the pre-delete values for surviving data.
- [ ] Compact-now dry run numbers equal the applied run's result.
- [ ] Timing at production scale: `vuln_ledger` wholesale rewrite and cold
      `api_bootstrap` stay inside the 6-min execution cap (Settings → Storage shows
      the cell budget; lower retention if approaching 6M cells).

## Quotas to know

- 6 min per execution (scans chunk with a 4.5-min budget per hop).
- UrlFetchApp: 20k/day consumer, 100k/day Workspace — a full 100k-finding scan is
  ~200 calls.
- Consumer accounts cap total trigger runtime at 90 min/day — deploy on a Workspace
  account for daily large scans (6 h/day).
- Spreadsheet ceiling: 10M cells; the ledger uses ~18 cells/vulnerability.
